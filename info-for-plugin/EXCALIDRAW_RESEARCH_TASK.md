## Excalidraw research task

Нужно провести техническое исследование возможностей интеграции Rolay plugin с установленным у пользователя Obsidian Excalidraw plugin.

Цель исследования:

- понять, реалистичен ли режим `single-editor live broadcast` для Excalidraw;
- не проектировать пока серверный протокол наугад;
- сначала выяснить, какие реальные hooks и API доступны в клиенте.

### Целевой UX, который хочется получить

Если это посильно:

- один пользователь редактирует drawing;
- остальные открывают тот же drawing в read-only/viewer режиме;
- viewers могут pan/zoom и вообще навигироваться по доске;
- viewers не могут изменять drawing;
- viewers видят live-изменения сцены от текущего editor;
- viewers видят курсор editor'а;
- в UI есть информация о том, кто сейчас editor, и действие для takeover/request control.

### Что нужно выяснить

Нужно не гадать, а проверить в коде/репозитории плагина Excalidraw и, если возможно, в локальном plugin repo:

1. Можно ли из нашего Obsidian plugin надёжно получить доступ к Excalidraw view / API для открытого drawing?
2. Есть ли стабильный способ:
   - читать текущую scene;
   - получать уведомления об изменении scene;
   - программно применять remote scene updates к уже открытому drawing?
3. Есть ли способ включить read-only / view-like режим так, чтобы пользователь:
   - не мог редактировать drawing,
   - но мог pan/zoom/навигацию?
4. Есть ли способ получить текущую позицию курсора editor'а или хотя бы scene pointer data?
5. Где лучше рисовать remote cursor:
   - через сам Excalidraw renderer,
   - через overlay поверх canvas,
   - или другим способом?
6. Не вызовут ли remote scene apply циклические локальные save/change events?
7. Как Excalidraw plugin работает с autosave и markdown persistence:
   - насколько файл является persistence layer,
   - можно ли обновлять live scene без прямого file reload?

### Что ожидается в результате

Нужен не общий рассказ, а конкретный инженерный вывод:

- какие гипотезы подтверждены;
- какие не подтверждены;
- какие конкретные API/классы/файлы ты нашёл;
- где есть риски хрупкой интеграции;
- какой из двух вариантов реалистичнее:
  - live scene broadcast to viewers;
  - fallback через save/autosave refresh.

### Формат ответа назад

После исследования:

1. Коротко напиши выводы в сообщении.
2. Если нужно передать серверной ветке дополнительный контекст, создай в plugin repo временную папку `info-for-server/`.
3. В ответе обязательно приложи промпт для этого серверного чата в codeblock.

Важно:

- сейчас приоритет не в немедленной реализации всей фичи, а в честном technical discovery;
- если по итогам исследования можно сразу сделать небольшой proof of concept, это хорошо, но только после того, как будут понятны реальные API;
- если выяснится, что полноценный live scene broadcast слишком хрупок, это тоже ценный результат.
